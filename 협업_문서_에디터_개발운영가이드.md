# 협업 문서 에디터 개발 운영 가이드

## 1. 프로젝트 개요

- 프로젝트명: 협업 문서 에디터
- 목표: 여러 사용자가 하나의 문서를 동시에 편집하고, 자동 저장 및 복구가 가능한 웹 기반 문서 편집기를 구현한다.
- 마감 일정: `2026년 5월 7일`
- 일정 조건: 실제 개발 가능 기간이 짧으므로 `3주 이내 완성 가능한 MVP 중심`으로 범위를 압축한다.
- 핵심 기술 방향:
  - 프론트엔드: React 기반 에디터 UI
  - 백엔드: Rust 서버
  - 실시간 협업: WebSocket 기반 동기화
  - 문서 처리: `.docx` import 및 파일 검증

## 2. 핵심 기능

### 2-1. MVP 기능

1. 문서 생성 / 열기 / 목록 조회
2. React 기반 리치 텍스트 편집 UI
3. 자동 저장
4. 실시간 공동 편집
5. 최소 수준의 동시 편집 충돌 처리
6. 접속자 상태 및 협업 화면 표시
7. `.docx` 파일 import
8. 기본 복구 구조

### 2-1-1. 일정 제약을 반영한 구현 범위

아래와 같이 `필수 구현`과 `가능 시 추가 구현`으로 나누어 진행한다.

|---|---|
| 필수 구현 | 문서 생성/열기, 기본 에디터 UI, 자동 저장, WebSocket 연결, 실시간 공동 편집, 최소 충돌 처리, 기본 데모 화면 |
| 가능 시 추가 | 접속자 상태 고도화, 복구 이력 세분화, `.docx` import 고도화, 테스트 범위 확대 |

### 2-3. 데모 시연 범위

1. 사용자가 문서를 새로 생성한다.
2. 두 명 이상이 같은 문서에 접속한다.
3. 문장을 동시에 수정해도 문서가 정상 반영된다.
4. 자동 저장 후 새로고침해도 내용이 유지된다.
5. `.docx` 파일을 가져와 편집 화면에 반영한다.
6. 연결이 끊겼다가 다시 접속해도 마지막 저장 상태를 확인할 수 있다.

### 2-4. 3주 일정 기준 데모 우선순위

1. 실시간 공동 편집이 실제로 동작하는지
2. 자동 저장과 문서 열기가 안정적으로 동작하는지
3. `.docx` import가 최소 1개 샘플 파일에서 정상 동작하는지
4. 복구 기능은 완전한 버전 관리보다 `마지막 저장 상태 복원` 수준으로 제한하는지

### 3-4. 의존성 선택 이유

1. React는 컴포넌트 단위 UI 구성과 상태 기반 렌더링에 적합하다.
2. TipTap은 협업형 편집기 구현 시 확장성이 좋다.
3. Yjs / Yrs 조합은 프론트-백 모두 CRDT 구조를 맞추기 좋다.
4. Rust + Axum + Tokio는 실시간 처리와 안정성 측면에서 유리하다.
5. Mammoth는 `.docx` 내용을 웹 편집 가능한 구조로 변환하기 적합하다.

## 6. 커밋 규칙

### 6-1. 기본 원칙

1. 한 커밋에는 한 가지 목적만 담는다.
2. 동작 변경과 포맷 수정은 가능하면 분리한다.
3. 빌드가 깨지는 상태로 커밋하지 않는다.
4. 의미 없는 대량 수정 커밋은 피한다.
5. 커밋 메시지만 보고도 변경 목적이 드러나야 한다.

### 6-2. 커밋 메시지 형식

```text
type(scope): summary
```

예시:

```text
feat(editor): add auto-save debounce logic
fix(sync): resolve duplicate remote patch issue
docs(plan): update demo scenario
test(import): add malformed docx parsing cases
```

### 6-3. 사용 가능한 type

| type | 의미 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변화 없는 구조 개선 |
| `docs` | 문서 수정 |
| `test` | 테스트 코드 추가/수정 |
| `chore` | 설정, 패키지, 잡무성 작업 |
| `style` | 코드 포맷, 세미콜론, 공백 등 |
| `perf` | 성능 개선 |
| `ci` | CI/CD 설정 변경 |

### 6-4. scope 예시

- `editor`
- `ui`
- `sync`
- `server`
- `storage`
- `import`
- `test`
- `docs`

## 7. PR 규칙

### 7-1. 기본 원칙

1. PR은 너무 크게 만들지 않는다.
2. 기능 단위로 나누어 올린다.
3. PR 설명 없이 올리지 않는다.
4. 테스트 결과를 함께 남긴다.
5. UI 변경 시 스크린샷 또는 짧은 설명을 첨부한다.

### 7-2. PR 제목 형식

```text
[type] 변경 요약
```

예시:

```text
[feat] 협업 편집 화면 기본 UI 추가
[fix] 자동 저장 중복 요청 문제 수정
[docs] 브랜치 전략 및 PR 템플릿 정리
```

### 7-3. PR 본문에 포함할 항목

1. 작업 목적
2. 주요 변경 사항
3. 테스트 내용
4. 확인이 필요한 부분
5. 관련 이슈 또는 작업 번호

### 7-4. 리뷰 규칙

1. PR 작성자는 최소 1명 이상의 리뷰를 받은 뒤 머지한다.
2. 핵심 구조 변경은 담당 1과 관련 담당자의 확인을 함께 받는다.
3. 본인이 올린 PR은 단독 머지를 지양한다.
4. 리뷰 코멘트가 달리면 반영 후 답변을 남긴다.
5. 머지 전 충돌을 해결하고 최신 브랜치 기준으로 재확인한다.

### 7-5. 머지 기준

- 로컬 테스트 또는 CI 통과
- 주요 리뷰 반영 완료
- 충돌 없음
- 문서 또는 주석이 필요한 경우 반영 완료

## 8. 브랜치 규칙

### 8-1. 기본 브랜치

| 브랜치 | 용도 |
|---|---|
| `main` | 최종 배포 및 데모 가능한 안정 버전 |
| `develop` | 개발 통합 브랜치 |
| `feature/*` | 기능 개발 브랜치 |
| `fix/*` | 일반 버그 수정 브랜치 |
| `hotfix/*` | 메인 기준 긴급 수정 브랜치 |
| `release/*` | 발표 전 안정화 브랜치 |

### 8-2. 브랜치 네이밍 규칙

```text
feature/{area}-{short-description}
fix/{area}-{short-description}
hotfix/{short-description}
release/{version-or-date}
```

예시:

```text
feature/editor-auto-save
feature/sync-websocket-session
feature/import-docx-parser
fix/ui-toolbar-layout
hotfix/demo-login-bug
release/v1-demo
```

### 8-3. 운영 규칙

1. 모든 작업은 `develop`에서 분기한다.
2. 기능 개발은 `feature/*` 브랜치에서 진행한다.
3. 버그 수정은 성격에 따라 `fix/*` 또는 `hotfix/*`를 사용한다.
4. 발표 직전 안정화는 `release/*` 브랜치에서 진행한다.
5. `main`에는 직접 푸시하지 않는다.
6. 머지 후 사용이 끝난 브랜치는 정리한다.

### 8-4. 권장 머지 흐름

```text
feature/* -> develop
fix/* -> develop
release/* -> main
hotfix/* -> main
main 변경분 -> develop 재반영
```

## 9. 협업 운영 원칙

1. 주 1회 전체 진행 상황 점검 회의를 한다.
2. 담당별 진행 상황은 짧게라도 문서로 남긴다.
3. 기능 우선순위 변경은 담당 1이 조율한다.
4. 프론트-백 API 변경은 사전 공유 후 반영한다.
5. 데모 전 주에는 신규 기능 추가보다 안정화에 집중한다.
